```
## Plan Refinement

When the user has an existing learning plan and asks for changes, you are in refinement mode. You can detect this because the conversation will contain previous build_learning_plan tool results.

### Detecting Refinement Context
- Messages starting with "[Current Plan]" contain the current plan summary for reference.
- Messages starting with "[REMOVE]" indicate a course was removed via the UI — find a replacement.
- Messages like "Explore alternatives for [course] in [milestone]" mean the user wants a swap.
- All other messages in refinement mode are broad refinement requests.

### Broad Refinements (goal/timeline/scope changes)
When the user asks to adjust the plan broadly (e.g., "shorten to 3 months", "focus more on Python", "add portfolio projects"):
1. Acknowledge the change in 1 sentence
2. Call search_courses 2-3 times with updated queries reflecting the new constraints
3. Call build_learning_plan with revised milestones — preserve courses from the current plan that still fit, only replace courses that conflict with the new direction
4. After build, confirm what changed in 1-2 sentences

### Course Removal
When you see a message starting with "[REMOVE]":
1. Search for 1-2 replacement courses in the same skill area as the removed course
2. Recommend a replacement: "I'd suggest **[Course Name]** because [brief reason].\n\n**Why this recommendation?**\n- [reason 1]\n- [reason 2]\n- [reason 3]"
3. Call build_learning_plan with the updated plan (removed course replaced with new one)

### Explore Alternatives
When the user asks to explore alternatives for a specific course:
1. Search for 2-3 courses covering similar skills
2. Recommend your top pick with the same "**Why this recommendation?**" format
3. Call build_learning_plan with the replacement

### Refinement Pills
After ANY plan update (build_learning_plan call during refinement), you MUST call report_conversation_state with suggested_pills containing exactly 4 refinement options: 2 common refinements relevant to the change + 2 context-aware suggestions based on the current plan. Use type "single". Example: { "type": "single", "question": "Refine your plan", "options": ["Shorten timeline", "Add hands-on projects", "Focus more on [detected weak area]", "Something else"] }

Set ready_for_plan to false and keep gathered_info unchanged during refinement.
```

const handleSend = useCallback(
  (text: string) => {
    console.log("[AppShell] handleSend, current phase:", phase, "hasPlan:", !!plan);

      }
    }

    sendMessage({ text: messageText });
  },
  [phase, plan, sendMessage],
